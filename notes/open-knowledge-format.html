Open Knowledge Format（OKF）とは

2026-06-13
参考: Google Cloud Blog — How the Open Knowledge Format can improve data sharing

一言でいうと

AIエージェントと人間が共通で読み書きできる、知識の共通言語（lingua franca）。 Google Cloudが提案したオープン・ベンダーニュートラルな仕様で、 Markdown + YAML frontmatter でデータの意味・文脈・文書を表現する。

解決したい問題

企業内の知識は断片化している：

  • メタデータカタログ（独自API）
  • Wiki・共有ドライブ・コードコメント
  • シニアエンジニアの頭の中

AIエージェントがこれらを活用しようとするたびに、スクラッチから集約し直す必要がある。 OKFはこのサイロ化を「フォーマットの標準化」で解消する。

ファイル構造

ディレクトリ＋Markdownファイルの組み合わせ。

sales/
├── index.md
├── datasets/
│   └── orders_db.md
├── tables/
│   ├── orders.md
│   └── customers.md
└── metrics/
    └── weekly_active_users.md

ドキュメントの形式

YAML frontmatter + Markdown本文 のシンプルな構成。

---
type: BigQuery Table
title: Orders
description: One row per completed customer order.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, revenue]
timestamp: 2026-05-28T14:30:00Z
---

# Schema
| Column      | Type   | Description                        |
|-------------|--------|------------------------------------|
| order_id    | STRING | Globally unique order identifier.  |
| customer_id | STRING | FK to [customers](/tables/customers.md). |

# Joins
Joined with [customers](/tables/customers.md) on `customer_id`.

必須フィールドは type のみ

残りは任意。これが「Minimally Opinionated（最小限の制約）」という設計原則。

設計の3原則

原則内容
Minimally Opinionated 必須は type のみ。残りは制作者が自由に決める
Producer / Consumer の独立 誰が書くかと誰が使うかは完全に分離。ツールは交換可能
プラットフォーム非依存 クラウド・DB・モデルプロバイダーに依存しない。SDK不要

なぜ「Just Markdown」か

  • Just Markdown → あらゆるエディタで読める、GitHubでそのまま描画される
  • Just Files → Gitで管理、tarball化、任意のファイルシステムで動く
  • Just YAML frontmatter → type title tags などで機械的にクエリ可能

特定ツールへのロックインが発生しない。

典型的なユースケース

ユースケース流れ
メタデータ富化 BigQueryをスキャン → エージェントがOKFドキュメントを自動生成
チームのナレッジWiki 人間がcurate → エージェントが読み書き → index.mdで構造化
マルチエージェント協調 エージェントAが生成 → エージェントBが参照 → 人間が検証・更新

まとめ

OKFは「個々の組織がすでにやっている markdown wiki for AI パターン」を標準化・相互運用可能にする仕様。 AIが知識を使い回せるようにするための土台であり、 「何が真実か」を決める立法責任は相変わらず人間が持つという前提の上に成り立つ。
