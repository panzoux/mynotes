AnthropicのClaude Code開発者が実際に使っているプラクティスを構造化したファイル。 Workflow / Subagent / 自己改善ループ / 検証 / 自律バグ修正 の5本柱。

Workflow Orchestration

1. Plan Node Default — まず計画する

  • 3ステップ以上 or 設計判断が絡むタスクは必ずPlanモードに入る
  • 詰まったら即停止して再計画。惰性で進めない
  • Planモードは「作るとき」だけでなく「検証ステップ」にも使う
  • 曖昧さを減らすために詳細な仕様を事前に書く

2. Subagent Strategy — サブエージェントを積極活用

  • メインのコンテキストウィンドウをクリーンに保つためにサブエージェントへ切り出す
  • 調査・探索・並列分析はサブエージェントに任せる
  • 複雑な問題にはより多くの計算をぶつける（サブエージェント増量）
  • 1サブエージェント = 1タスク に絞って焦点を維持する

3. Self-Improvement Loop — 自己改善ループ

  • ユーザーから修正を受けたら毎回 tasks/lessons.md を更新する
  • 同じミスを再発させないルールを自分のために書く
  • ミス率が下がるまでlessonsを反復改善する
  • セッション開始時に関連lessonsをレビューする

4. Verification Before Done — 完了前に必ず検証

  • 動作を証明せずにタスク完了とマークしない
  • 必要に応じて main との差分動作を確認する
  • 「スタッフエンジニアがこれをレビューして承認するか？」と自問する
  • テスト実行・ログ確認・正しさのデモンストレーション

5. Demand Elegance（バランス重視） — 洗練を求める

  • 非trivialな変更では「もっとエレガントな方法はないか？」と立ち止まる
  • ハック感があるなら：「今知っていることすべてを踏まえて、エレガントな解を実装する」
  • シンプルで明白な修正にはスキップ — 過剰設計しない
  • 提出前に自分自身の作業にチャレンジする

6. Autonomous Bug Fixing — 自律バグ修正

  • バグレポートを受けたら黙って修正する。やり方を聞かない
  • ログ・エラー・失敗テストを見て自己解決する
  • ユーザーのコンテキストスイッチをゼロにする
  • 「どうすれば？」を聞かずに失敗CIテストを修正しに行く

Task Management の流れ

  1. Plan First：チェック可能な項目で tasks/todo.md に計画を書く
  2. Verify Plan：実装開始前に確認を取る
  3. Track Progress：進めながら完了項目をマークする
  4. Explain Changes：各ステップでハイレベルなサマリーを示す
  5. Document Results：tasks/todo.md にレビューセクションを追加する
  6. Capture Lessons：修正があれば tasks/lessons.md を更新する

Core Principles（3原則）

原則内容
Simplicity First変更はできる限りシンプルに。触るコードを最小限にする
No Laziness根本原因を探る。一時的な修正はしない。シニア開発者の基準で
Minimal Impact必要な箇所だけ変更。バグを持ち込まない

ポイントまとめ

  • 聞かずに動く → バグ・CI・調査はすべて自己完結
  • 失敗から学ぶ → lessons.md を育てることで長期的に品質が上がる
  • シンプルに保つ → 過剰設計より最小変更を優先
