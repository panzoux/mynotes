AI時代の文書信頼性問題と属性・仕組みの設計

2026-06-13
参考: Zenn — ACN Tech Blog

問題の本質

AIが文書を大量生成できるようになったことで、「書いてある＝信頼できる」という暗黙のシグナルが消滅した。
以前は文書の量が少なく人間が選別できたが、今は量が選別能力を超え、存在自体がシグナルを発さなくなった。

二種類の病

病名症状原因
WIP病 未承認の文書が「真」として扱われる 承認状態が機械可読な形で記録されていない
腐敗病 承認後も文書が現実と乖離したまま放置される 維持する仕組みがなく、自然に腐る

文書に必要な属性

文書を生成した時点で、機械可読な形で付与すべきメタデータ：

status:           approved / draft / 要再承認
owner:            責任者（人名 or チーム）
source_of_truth:  true / false
last_validated_at: 最終検証日時
depends_on:       [参照先のコード・仕様 etc.]

source_of_truth: false の文書は「参考情報」として扱い、意思決定の根拠にしない。 depends_on で参照先が変わったら自動的に status: 要再承認 に落とす。

仕組みの構造（三権分立モデル）

役割担当フェーズ内容
立法（上流） 承認できる粒度を決める 「何を1文書とするか」を人間が設計する
採決（中流） 承認の儀式 レビュー・承認のプロセス実行
司法（下流） 区分を検証・結線 依存関係の変化を検知して自動的に鮮度を落とす

差別化されるのは上流（粒度の設計）と下流（自動維持メカニズム）。 中流の採決はAIが補助できるが、立法責任は人間に残る。

核心的な制約

AIは「何を真とするか」の立法責任を担えない。
最後の引き受けは必ず人間に残る。 文書管理の設計とは、AIに任せる部分と人間が持つ部分を明確に切り分けることである。
